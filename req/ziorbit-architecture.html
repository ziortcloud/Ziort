ZiOrbit Platform Architecture

The Two-Layer
Platform Vision

A marketplace network on top + intelligent inter-app data flow underneath.

What you're building is not just SaaS — it's a platform economy

Most SaaS tools are islands. ZiOrbit becomes a gravitational platform — every business that joins pulls others in. A ZiRetail shop needs suppliers → ZiSupply connects them. A ZiFleet owner needs drivers → ZiPilot connects them. A factory needs transport → ZiLoad matches them. The more businesses join, the more valuable the network becomes for everyone. This is the model that made Alibaba, Uber, and Zomato platform giants.

Layer 1: ZiNetwork

The marketplace layer. Businesses and individuals post needs, offers, and requirements. Others respond, match, transact. Supply meets demand across all verticals.

Layer 2: Inter-App

The data layer. ZiPawn talks to ZiLedger. ZiFleet syncs with ZiLoad. ZiRetail connects to ZiSupply. One customer profile, one auth, data flows intelligently across apps.

Layer 2 — Intelligence

Inter-App
Event Bus

How ZiOrbit products talk to each other automatically — no manual copy-paste, no data silos.

The principle: One action triggers the right reaction

When ZiPawn marks a loan as overdue, ZiPost automatically sends a reminder — without the user doing anything. When ZiRetail stock drops below threshold, ZiNetwork posts a requirement. When ZiLoad completes a delivery, ZiLedger records the expense. This is the intelligence layer — events flow between apps through a central event bus.

Key Inter-App Events — Full Map
── INTER-APP EVENT CONNECTIONS ────────────────────── ZiPawn loan.due_reminder → ZiPost.send_whatsapp loan.overdue → ZiPost.send_sms + ZiLedger.flag_receivable item.unclaimed → ZiNetwork.post_auction payment.received → ZiLedger.record_income ZiRetail stock.low → ZiNetwork.post_requirement (auto) sale.completed → ZiLedger.record_income order.placed → ZiLoad.create_delivery_job customer.new → ZiPost.send_welcome ZiLedger payment.overdue → ZiPost.send_reminder expense.recorded → ZiBooks.sync_entry report.monthly → ZiPost.send_summary_to_owner ZiLoad job.created → ZiFleet.notify_vehicles job.accepted → ZiPilot.assign_driver delivery.confirmed → ZiPay.release_payment delivery.confirmed → ZiLedger.record_expense ZiFleet vehicle.idle → ZiNetwork.post_availability trip.started → ZiPilot.notify_driver maintenance.due → ZiPost.alert_owner ZiCare appointment.booked → ZiPost.send_confirmation appointment.tomorrow → ZiPost.send_reminder stock.low → ZiNetwork.post_requirement payment.collected → ZiLedger.record_income ZiHR salary.due → ZiPay.initiate_payroll attendance.done → ZiLedger.record_salary_expense staff.birthday → ZiPost.send_greeting
How the Event Bus Works — Technical
── IMPLEMENTATION: Supabase Realtime + Edge Functions ─ 1. An app writes an event to the events table: INSERT INTO zi_events ( source_app, -- 'zipawn' event_type, -- 'loan.due_reminder' tenant_id, -- the business payload, -- { loan_id, customer_phone, amount } created_at ) 2. Supabase Realtime triggers an Edge Function 3. Edge Function reads the event_routing table: zi_event_routing event_type: 'loan.due_reminder' target_app: 'zipost' action: 'send_whatsapp' enabled: true ← tenant can toggle ON/OFF 4. Edge Function calls ZiPost API with payload 5. ZiPost sends WhatsApp via MSG91 6. Event marked complete, logged for audit Result: ZiPawn user never configured ZiPost. It just works. Automatically.
User Control — Integration Settings Panel

Every tenant controls their own integrations

In ZiOrbit settings, each business sees a simple panel: "When ZiRetail stock is low → Auto-post to ZiNetwork: ON/OFF". They can enable or disable each inter-app connection. Power users can customize triggers. Basic users just leave defaults on. This is the difference between magic and chaos.

Data Architecture

Core Schema
Design

The key database tables that power ZiNetwork and the inter-app layer.

zi_network_posts — The marketplace feed
id uuid PRIMARY KEY tenant_id uuid → tenants.id post_type enum -- requirement|supply|load_job|gig|partnership|auction source_app text -- 'ziretail'|'zipawn'|'zicare' etc title text description text category text -- 'grocery'|'medical'|'transport'|'labour' location_city text location_state text quantity numeric unit text -- 'kg'|'tonnes'|'units'|'trips' budget_min numeric budget_max numeric deadline timestamptz status enum -- open|matched|in_progress|completed|cancelled visibility enum -- public|verified_only|invite_only auto_generated boolean -- true if posted by inter-app event created_at timestamptz
zi_network_responses — Bids and offers on posts
id uuid PRIMARY KEY post_id uuid → zi_network_posts.id responder_id uuid → tenants.id response_type enum -- quote|bid|application|counter_offer message text offered_price numeric delivery_date date status enum -- pending|accepted|rejected|withdrawn created_at timestamptz
zi_events — Inter-app event bus
id uuid PRIMARY KEY tenant_id uuid → tenants.id source_app text -- 'zipawn'|'ziretail' etc event_type text -- 'loan.due_reminder'|'stock.low' payload jsonb -- event-specific data status enum -- pending|processing|completed|failed processed_at timestamptz created_at timestamptz
zi_event_routing — What triggers what (tenant-configurable)
id uuid PRIMARY KEY tenant_id uuid → tenants.id -- null = platform default event_type text -- 'loan.due_reminder' target_app text -- 'zipost' action text -- 'send_whatsapp' enabled boolean -- tenant can toggle config jsonb -- delay_hours, template_id etc
zi_pilot_profiles — Driver / gig worker profiles
id uuid PRIMARY KEY user_id uuid → auth.users.id worker_type enum -- driver|delivery|helper|captain license_number text license_type text -- LMV|HMV|Transport vehicle_types text[] -- ['truck','tempo','bike'] base_city text available boolean rating numeric total_trips integer verified boolean -- KYC + license verified wallet_balance numeric -- ZiPay wallet

Implementation Strategy

How to Build
This

Exact sequence. What to build first. What to defer. No wasted effort.

Phase 1 — Now
Build ZiNetwork as a simple feed (MVP)
One table: zi_network_posts — any ZiOrbit user can post a requirement or supply offer
Simple feed UI in ZiOrbit dashboard — filterable by category and city
ZiPost sends WhatsApp notification to matching businesses when a new relevant post appears
Responses via WhatsApp or in-app — keep it simple, no complex bidding yet
Do NOT build ZiFleet/ZiLoad/ZiPilot yet — just let users post transport needs manually on ZiNetwork
Phase 1 — Now
Build the Event Bus (just 3 events to start)
Create zi_events table and Supabase Edge Function listener
Event 1: ZiPawn loan.due → ZiPost sends WhatsApp reminder. This alone saves pawn brokers hours every week.
Event 2: ZiLedger payment.overdue → ZiPost sends follow-up SMS
Event 3: ZiRetail stock.low → ZiNetwork auto-posts requirement
These 3 events prove the concept and create immediate user value — expand later
Phase 2 — After ZiPartner Live
Build ZiLoad + ZiFleet (linked products)
Build ZiFleet first — vehicle registration, basic trip management, earnings tracking
Build ZiLoad second — load posting, vehicle matching from ZiFleet pool, delivery tracking
Connect ZiLoad jobs to ZiFleet via the event bus: load.posted → fleet.notify_available_vehicles
ZiNetwork becomes the public load board — ZiLoad is the management backend for matched parties
Phase 2
Build ZiPilot — driver/gig profiles
Driver profile: license upload, KYC, vehicle types, base city, availability toggle
Job feed: drivers see available trips from ZiLoad + gig posts from ZiNetwork
Earnings wallet: ZiPay integration for automatic payment on trip completion
Rating system: both driver rates business, business rates driver — builds trust
Phase 3 — Scale
Full ZiNetwork marketplace + advanced routing
Add bidding, counter-offers, escrow payments through ZiPay
Verified supplier badges — paid tier for distributors and wholesalers
AI matching — ZiOrbit suggests the best supplier/vehicle/driver based on history
Expand ZiPilot to non-transport gigs — skilled labour, delivery, event staff
International expansion — ZiNetwork becomes cross-border B2B trade platform
